File-System Structure

  • 文件结构
    • 逻辑储存模块
    • 相关信息的容器
  • 二级存储（disks）
  • 分层管理
  • FCB(文件控制块)：包含文件管理的信息

Layered file system

分层的目的：每层设计利用更低层的功能，创建新的功能，以用于更高层服务。

I/O control

包括设备驱动程序和中断处理程序，以在主内存和磁盘系统之间传输信息。设备驱动程序可以作为翻译器。

basic file system

只需向适当设备驱动程序发送通用命令，以读取和写入磁盘的物理块。每个物理块由磁盘的数字地址来标识（如，驱动器：1，磁道：3）。

也管理内存缓冲区和保存各种文件系统、目录和数据块的缓存。

file-organization module

知道文件及其逻辑块以及物理块。可以将逻辑块地址转为物理块地址，以供basic file system传输。

Logical file system

管理元数据信息。元数据包括文件系统的所有结构，而不包括实际数据（或文件内容）。逻辑文件系统Logical file system管理目录结构，以便根据给定文件名称为文件组织模块提供所需的信息。

它通过文件控制块来维护文件结构。文件控制块（PCB)包含有关文件的信息，包括所有者、权限、文件内容的位置等。

文件系统实现

概述

文件名称保留在目录信息中。

实现文件系统的数据结构

  • disk structure
    • boot control block
    • Volume control block(superblock in Unix)
    • Directory structure per file system
    • Per-file PCB(inode in Unix)
  • in-memory structures
    • In-memory mount table about each mounted volumn
    • 目录缓存
    • 打开文件表
  • 先经过进程的打开文件表，（通过文件描述符作为下标），找到系统打开文件表

Virtual File System

  • 虚拟文件系统提供一种面向对象的方式实现文件系统
  • 允许同一个系统调用接口（API)来操作不同的文件系统

目录实现

线性列表

编程简单，执行费时

哈希表

根据文件名称获得一个值，返回线性列表内的一个元素指针。

可以采用溢出链接的哈希表：每个条目可以是链表而不是单个值

分配方法

连续分配

  • 要求：每个文件在磁盘上占有一组连续的块。

  • 文件的连续分配可以用首块的磁盘地址和连续的块数来定义。

  • 访问非常容易，支持顺序访问和直接访问。（对于直接访问一个文件的从块b开始的第i块，可以直接访问块b+i.

  • 一个难题是：为新文件找到空间。动态存储分配（容易造成外部碎片）

  • 另一个难题是：确定一个文件需要多少空间。

Extent-Based Systems

通过拓展来分配磁盘块

每个拓展是磁盘上的连续块

链表分配

  • 需要保留一个指针，指向下一个block

  • Simple -need only starting address
  • 没有浪费的空间
  • No random access,poor reliability
  • Mapping（第一个word放指向下一块的指针）

File-Allocation Table

mini-link:FAT每一项是一个word，存有指向下一块的位置。

使用内存块的指针链表代替磁盘的指针链表。

FAT表磁盘贮存，mount后加载到内存。

缺陷：如果一个pointer出错，可能导致后面的信息全部丢失

索引分配

  • Brings all pointer together into the index block
  • Logical views

缺点：需要多访问一次磁盘（索引块）

通常来说：会采用组合的模式：既有直接访问，也可以有多级访问

Free-Space Management

空闲块的管理

  • Bit map requires extra space
    • Example:
      • block size : 2122^{12}212bytes
      • disk size: 2302^{30}230bytes
      • n = 230/212=2182^{30}/2^{12} = 2^{18}230/212=218bits(32 K bytes)
  • Easy to get contiguous files
  • Linked list(free list)
    • Cannot get contiguous space easily
    • But basically can work (FAT)
    • No waste of space
  • Grouping - a modification of the Linked list
    • Addresses of the n free blocks are stored in the first block.
  • Counting
    • Address of the first free block and number n contiguous blocks.

Need to protect

  • 空闲链表的指针
  • bit map
    • Must be kept on disk
    • The copy in memory and disk may differ
    • Cannot allow for block[i] to have a situation where bit[i] = 1 in memory and bit[i] = 0 on disk
  • Solution:
    • 先给硬盘上的 bit[i]置1
    • 再释放block[i]
    • 最后改内存上的bit[i]

（1）

  • 连续分配：理论上不受限制，可大到整个磁盘文件区
  • 链接分配：由于块的地址为4字节，所以最多的链接索引块数为2322^{32}232，而每个盘块中存放文件大小为4092B。链接分配可管理的最大文件为:4G×4092B=16368GB4G \times 4092B = 16368GB4G×4092B=16368GB
  • 链接索引：由于块的地址为4字节，所以最多的链接索引块数为2322^{32}232,而每个索引块中有1023个文件块的地址，盘块大小为4B。假设最多有n个索引块，则1023×n+n=2321023\times n+n=2^{32}1023×n+n=232,最大文件为:4G×4092B=16368GB4G \times 4092B = 16368GB4G×4092B=16368GB
  • 二级索引：由于盘块大小为4KB，每个地址用4B表示，一个盘块可存1K个索引表目。
    • 最大文件容量4KB×1K×1K=4GB4KB\times 1K \times 1K=4GB4KB×1K×1K=4GB
  • LINUX 混合分配：LINUX的直接地址指针有12个，还有一个一级索引，一个二级索引，一个三级索引
    • 48KB+4MB+4GB+4TB48KB + 4MB + 4GB + 4TB48KB+4MB+4GB+4TB

（2）

  • 连续分配：不管大小文件，只需要在文件控制块FCB设两项，一是首块物理块块号，另一是文件总块数，不需专用块来记录文件的物理地址。
  • 链接分配：对大小文件都只需在文件控制块FCB中设二项，一是首块物理块块号，另一是末块物理块块号；同时在文件的每个物理块中设置存放下一个块号的指针。
  • 一级索引：
    • 对20KB小文件只有5个物理块大小，所以只需一块专用物理块来做索引块，用来保存文件的各个物理块地址；
    • 对于20MB大文件共有5K个物理块，由于链接索引的每个索引块只能保存（1K-1)个文件物理块地址（另有一个表目存放下一个索引块指针），所以他需要6块专用物理块来作链接索引块。
  • 二级索引：
    • 对20KB小文件用一个物理块作第一级索引，另一块作二级索引；
    • 对20MB大文件，用一块物理块作第一级索引，另外需要5块物理块（一块存1K地址项，每一项指向一块能存4KB空间，故一块可以指代4MB空间）作第二级索引。
  • LINUX的混合分配：
    • 对20KB小文件只需在文件控制块FCB的i_addr[15]中使用前5个表目存放文件的物理块号，不需专用物理块。
    • 对20MB小文件，（12∗4KB=48KB12*4KB = 48KB12∗4KB=48KB)，用一级索引一块保存大文件接着的1K块块号（1K∗4KB=4MB1K*4KB = 4MB1K∗4KB=4MB）,剩下还有接近16MB，还要用二级索引存大文件以后的块号，一块物理块作为二级索引，第二级索引4块（4KB×1K×4=16MB4KB\times 1K \times 4 = 16MB4KB×1K×4=16MB）。

（3）

  • 连续分配：读前面第5.5KB的信息，只需要一次IO操作；读后面第（16MB+5.5KB）的信息，也只需要一次。
    • 未读大文件前面和后面信息都需先计算信息在文件中的相对块数，前面信息相对逻辑块号为 5.5K/4K=15.5K/4K =15.5K/4K=1,后面信息相对逻辑块号（(16M+5.5K)/4K = 4097)
  • 链接分配：读前面第5.5KB的信息，需要两次IO；读后面（(16MB+5.5KB)/ 4092 = 4102）的信息需要4103次IO
  • 链接索引：读前面第5.5KB信息，需要首先读取索引块，再获取第一块的信息，即两次IO；读后面第（16MB+5.5KB）的信息，需要读取第四块索引块（0,1,2,3,4 -> 5次IO），在读取所需的信息，一共6次。
  • 二级索引：为读大文件前面和后面信息的操作相同，需要一次第一级索引，一次第二级索引，最后一次所在数据块。3次IO。
  • LINUX混合分配：读前面第5.5KB的信息（在内存文件控制块FCB的i_addr第二个表目中读取信息所在块块号），只需要一次IO操作。后面(16MB+5.5KB)的信息在二级索引，所以需要3次IO.
